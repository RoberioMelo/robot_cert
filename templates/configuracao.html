1. Chave API (segurança)

Se o ficheiro .env do FastAPI tiver a variável API_KEY=..., todas as rotas /api/... exigem o cabeçalho X-API-Key com o mesmo valor. O agente usa CERT_ROBOT_API_KEY no .env (mesmo texto). Aqui o valor fica apenas no seu browser (localStorage), nunca no servidor.

Campo e guardar

2. Pastas no Windows (origem e destino)

Estes caminhos são a referência que o GET /api/settings devolve. O agente lê o endpoint e procura ficheiros .pfx na pasta de origem; os vencidos podem ser movidos para a pasta de vencidos. Use caminhos absolutos (ex. C:\Certs).

  • Se deixar vazio, o servidor aplica o padrão de CERT_SOURCE_DIR / CERT_EXPIRED_DIR do .env (útil se o API correr no mesmo PC).
  • Com Supabase ligado, estes campos vão para a tabela; sem Supabase, guardam em data/portal_settings.json.

Valores e ID da máquina

O ID identifica a máquina nas ingestões e nos comandos remotos (abaixo). Deve coincidir com o que o agente envia.

3. Comandos remotos (fila no agente)

Pode enfileirar ações. O run_agent.py, em cada ciclo, chama GET /api/agent/next?machine_id=... e, se houver fila, executa: por exemplo, mover vencidos no disco do Windows onde o agente corre (não no servidor, se forem máquinas diferentes). Use o mesmo ID da máquina que na secção anterior; * aplica a qualquer agente.

Desative o questionário com POLL_COMMANDS=0 no .env do agente.
